# Claude Code 仓库上下文机制启示

这份备忘记录从 `claude-code/` 源码中观察到的上下文组织方式，以及它对后续“面向文档库问答智能体”的设计启发。

## 关键观察

Claude Code 并不预先把整个代码仓库内容做语义索引后塞进模型上下文。它采用的是按需探索模式：

- 启动时只注入轻量仓库状态，例如当前分支、main 分支、简短 git status、最近 commit，以及 `CLAUDE.md`/规则文件等项目说明。
- 仓库内容通过工具按需获取：`Glob` 找文件路径，`Grep` 用 ripgrep 搜内容，`Read` 精确读取文件或行号范围，`LSP` 提供定义、引用、符号和 hover 信息。
- 文件路径索引用于路径补全和快速找文件，主要索引的是路径列表，不是文件全文内容。
- 用户 `@` 文件、IDE 当前选区/打开文件、刚编辑文件的 diff、诊断信息、任务状态、嵌套 `CLAUDE.md` 规则等，会以 attachment 的形式进入下一轮上下文。
- 长会话通过 tool result 预算、microcompact、context collapse、autocompact 等机制续航，压缩摘要会刻意保留用户意图、关键文件、代码片段、错误修复和当前工作。
- 对开放式探索或会产生大量中间输出的任务，它倾向用子代理/fork，把探索噪音隔离在子上下文里，主上下文只接收最终摘要。

## 设计原则

1. 不要把“理解仓库”设计成一次性全量灌入上下文。更稳的方式是让模型拥有可靠的检索工具，并把每次检索结果控制在可读、可引用、可追溯的范围内。
2. 上下文应该分层：长期规则、当前任务、最近读过/改过的材料、外部检索结果、压缩后的历史摘要，各层生命周期不同。
3. 搜索工具的输出默认应该轻量：先返回路径、标题、命中行、片段和计数，只有在模型需要时再读取完整内容。
4. 重复读取同一内容应有缓存或“未变化”占位，避免把相同文本反复注入上下文。
5. 对长任务，要把“中间探索噪音”和“可继续工作的结论”分开。子代理适合做广泛调查，主代理负责综合判断和最终行动。
6. 压缩摘要不能只写闲聊摘要，必须保留可恢复工作的硬信息：文件路径、段落位置、关键原文片段、用户要求、已排除方案、待办事项。

## 对文档库问答智能体的启发

文档库和代码仓库不同。代码仓库天然适合 `grep`、路径、符号、LSP；文档库更依赖语义检索、结构化元数据和可引用出处。因此可以借鉴 Claude Code 的“按需上下文”思想，但检索层需要换成面向文档的实现。

建议的文档库问答架构：

- 文档预处理：保留 MinerU 输出的 Markdown、图片目录，并为每份文档记录来源路径、类别、标题、日期、发文机关、文号、页码/图片引用等元数据。
- 分块索引：按标题、段落、表格、页码进行 chunk，避免固定长度切碎公文结构。
- 混合检索：结合关键词检索、向量检索、元数据过滤和必要的重排。公文问答里，文号、人名、日期、机关名称等精确字段很重要，不能只靠 embedding。
- 上下文装配：默认只注入命中文档的片段、标题、路径和来源，不直接注入整篇文档。需要全文时再按文档路径读取。
- 引用约束：回答时带出处，例如文档名、章节/段落、必要时页码或图片引用。
- 会话记忆：记录用户当前关注的主题、已确认的文档范围、已经引用过的文档，避免重复检索和重复注入。
- 长问答压缩：压缩时必须保留用户问题、已检索文档、采用/排除的证据、尚未回答的点，而不是只总结对话。
- 子任务隔离：对“请梳理所有相关文件”“比较多个年度政策变化”这类任务，可用子代理分别检索不同类别/年份，主代理只合并结论和证据。

## 初步实现切入点

先做一个最小可用闭环：

1. 扫描 `documents_mineru/**/*.md`，生成文档清单和元数据。
2. 将 Markdown 按公文结构分块，保存 chunk、来源路径、标题层级、类别和相邻上下文。
3. 提供 `search_documents` 工具：支持关键词、向量、类别/日期/机关过滤，返回轻量命中片段。
4. 提供 `read_document` 工具：按文档路径、标题或 chunk id 读取更大范围内容。
5. 在 agent 中加入上下文预算控制：检索先轻后重，回答必须引用来源。
6. 后续再加入重排、表格结构化、图片 OCR 引用和多文档比较。

一句话总结：Claude Code 的核心启发不是“做一个更大的索引”，而是“让模型在有限上下文中持续、可控、可追溯地取用外部知识”。文档库问答也应该沿这个方向设计。
